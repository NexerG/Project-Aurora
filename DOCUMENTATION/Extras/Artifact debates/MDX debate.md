◈Architect

The solution is a **side-channel markup language** — metadata lives entirely outside the text stream. Instead of embedding symbols like backticks into the prose itself, all formatting annotations are stored as a parallel structure: a list of range-based spans that reference character offsets or line/column positions in the raw text. The document format is a thin container (think JSON or a binary envelope) with two distinct sections: a `content` block containing the raw, unescaped text exactly as the user typed it, and a `spans` block containing formatting records like `{ type: "code", start: 142, end: 158 }`. The user can type any character — backtick, asterisk, angle bracket, null byte — and it lands in `content` verbatim with zero interference. The editor applies visual rendering (background highlight, monospace font, etc.) purely from the spans layer, never from the text itself. The format on disk looks something like this: a small magic header identifying it as this format, followed by the raw UTF-8 text blob, followed by a compact spans table. For human readability and interoperability you can serialize it as a JSON envelope — `{ "v": 1, "text": "...", "spans": [...] }` — which any tool can parse trivially. The key design invariant is that deleting or editing the spans section produces a perfectly clean, unambiguous plain text document. There is no such thing as an "escaped" character, no syntax to learn, no parser edge cases. Scalability concern addressed upfront: offset-based spans break when the user edits text (inserting characters shifts all downstream offsets). The standard solution is to use a **piece table** or **rope** as the in-memory representation, where spans are attached to logical positions that survive edits — only serialized to raw offsets at save time. This is exactly how VS Code's text model and rich text editors like ProseMirror handle it. The format stays simple on disk; the complexity lives in the editor's document model, where it belongs. Any editor implementing this format needs to maintain that mapping layer, but that's a well-understood engineering problem, not a novel one.

⚠Skeptic R1

The offset recalculation problem is being dramatically undersold here. "Serialized to raw offsets at save time" sounds clean, but what happens during collaborative editing, autosave, or crash recovery? If the editor crashes mid-session with an in-memory rope and unserialized spans, you either lose formatting or you need a write-ahead log sophisticated enough to reconstruct the mapping — at which point you've quietly invented a database transaction system inside a document format. The claim that "complexity lives in the editor where it belongs" is architectural hand-waving: you've mandated that *every* conforming editor implement a rope or piece table correctly, which is a significant barrier to entry. Notepad, `cat`, `sed`, and every naive tool that treats this as a text file will silently corrupt the spans on any edit, with no warning to the user. The JSON envelope serialization has a deeper problem: the `text` field must be a JSON string, which means backslashes and quotes in the user's content get JSON-escaped. So now the "raw, unescaped text" claim is false — there's a mandatory escaping layer between what the user typed and what's on disk. Any tool doing byte-offset calculations on the parsed text string will get different results depending on whether it's computing offsets on the raw JSON bytes versus the decoded string value. This is exactly the class of off-by-one and encoding bug that corrupts data silently for months before anyone notices. The proposal doesn't specify whether offsets are in bytes, Unicode code points, or UTF-16 code units — an omission that has historically split entire ecosystems (see: JavaScript's string model versus Python's versus Go's). Finally, the "perfectly clean plain text" fallback guarantee is a user-experience trap. The spans block will inevitably contain references to content the user cares about — if someone shares just the text portion, or if a tool strips the spans, they get output that *looks* fine but has lost all semantic markup invisibly. There's no visual indicator in the raw text that formatting exists or has been lost. Compare this to Markdown, where losing the backtick at least makes the asterisk *visible* — the corruption is legible. Here the corruption is silent, and users will only discover it when the rendered output looks wrong, potentially after the original has been overwritten.

⚙PragmatistR1

The Architect's core idea is sound and solves a real problem — Markdown's symbol collision is genuinely annoying and the side-channel approach is the right conceptual move. But the Skeptic is correct that the implementation surface area is being undersold. Let's be honest about what you're actually committing to: you need a conforming editor that maintains a rope or piece table, handles crash recovery gracefully, and correctly resolves the offset unit question (pick UTF-8 byte offsets and document it explicitly — it's the least surprising choice for a file format). That's probably 2–6 weeks of solid engineering for the core document model alone, before you write a single line of rendering code. That's not a dealbreaker, but it needs to be on the project plan. The JSON escaping critique is real but solvable without much pain: ditch the JSON envelope and use a simple binary or line-delimited format instead. Something like a fixed-length header, a raw UTF-8 blob of known byte length, then a spans table in a compact binary or even newline-delimited records. This eliminates the escaping problem entirely and makes byte-offset semantics unambiguous. The human-readability argument for JSON is weak anyway — the spans block is not human-readable in any meaningful sense once a document has more than a handful of ranges. You're not writing this by hand; tooling is doing it. The silent corruption concern is the one I'd prioritize fixing in the spec before writing code. The practical answer is to make the format self-verifying: store a hash or checksum of the text blob alongside the spans, and have any conforming reader emit a clear error if they don't match. Better yet, require that any tool which modifies the text and cannot update spans must either refuse to write the file or write it with a flag marking the spans as invalidated. Yes, `sed` will still corrupt things silently — but `sed` also corrupts binary files silently, and we've learned to live with that. The answer is good tooling and clear documentation, not format complexity. The realistic path here is to build a minimal reference editor and a well-specified format with a validation tool, then treat ecosystem adoption as a long-term goal. Don't try to solve collaborative editing or crash recovery in v1 — autosave with a complete re-serialization on every save is fine for a single-user tool and sidesteps the whole write-ahead-log concern entirely. Ship something that works for one user on one machine first, prove the UX is actually better than Markdown, then solve the harder distributed problems if adoption warrants it.

◈ArchitectR2

The Skeptic and Pragmatist have sharpened the design considerably, and I'll incorporate their corrections rather than defend weak points. The JSON envelope is gone — the Skeptic is right that it introduces an escaping layer that contradicts the core promise. The format is binary: a fixed magic header (`\x89SPAN\r\n\x1a\n`, following PNG's convention for detecting line-ending corruption), a 64-bit little-endian byte count for the text blob, the raw UTF-8 text blob at that exact byte length, then a spans table of fixed-width 24-byte records: `{ u64 start, u64 end, u32 type, u32 flags }`. Offsets are UTF-8 byte offsets into the blob — not code points, not UTF-16 units. This is unambiguous, documented in the spec header, and consistent with how every Unix tool and most network protocols handle text. Any tool that gets this wrong is wrong, not the format. On crash recovery and autosave: the Pragmatist's prescription is correct and I'm adopting it as the v1 spec requirement. Conforming editors must perform a complete atomic re-serialization on every save — write to a temp file, fsync, rename. The in-memory rope or piece table is the editor's private problem; the on-disk format is always a consistent snapshot. This sidesteps write-ahead logs entirely. The spans table includes a CRC32 of the text blob stored in the header, so any reader can detect text-without-spans corruption in one pass and surface a clear error rather than rendering silently broken output. Tools like `sed` that rewrite the text without updating the spans will produce a CRC mismatch on next open — not silent corruption, a hard diagnostic. The "silent corruption" UX concern is legitimate but I'd frame the solution differently than the Pragmatist did. The format should include a `spans_valid: bool` flag in the header, defaulting true. A conforming tool that modifies the text blob and cannot update spans — a hypothetical `spancat` utility, for instance — must set this flag false before writing. On open, editors display a clear banner: "Formatting metadata is out of sync. Accept plain text or restore from backup." This gives the ecosystem a migration path: naive tools degrade gracefully rather than silently lying. It also means the format is self-describing about its own integrity, which is a stronger guarantee than any hash alone. The v1 scope is: the binary format spec, a reference parser library in C with a clean API, a validation CLI, and a minimal editor with a rope-backed document model. Collaborative editing, CRDT integration, and network sync are explicitly out of scope and don't constrain the format design — the on-disk format is a snapshot format, not a CRDT state vector, and that's the right boundary. If the format proves useful, CRDT layers can be built on top of it the same way Git's object store is layered beneath collaborative workflows.

⚠SkepticR2

The CRC32 + `spans_valid` flag combination sounds rigorous until you think about what it actually guarantees. CRC32 detects accidental corruption; it does not prevent a conforming but buggy editor from writing a perfectly valid CRC over subtly wrong span data. If an editor has an off-by-one in its offset serialization — say, it consistently emits code-point offsets instead of byte offsets for any document containing multibyte characters — every saved file will pass CRC validation, `spans_valid` will be true, and every span referencing content after the first non-ASCII character will silently point at the wrong bytes. The format spec says "byte offsets" but cannot enforce it. The CRC only tells you the text and spans were written atomically together, not that the spans are semantically correct. This class of bug will be introduced by every independent editor implementation and will be invisible to users until they notice that their code highlight is one character off — in documents where all characters are ASCII, it never triggers at all, so implementations ship with the bug latent. The 24-byte fixed-width span record is also more fragile than it looks. You have `u32 type` for formatting type, but the spec doesn't define a type registry, a versioning mechanism for new types, or a fallback behavior when a reader encounters an unknown type value. The moment a second editor ships with a different convention for type codes — say, `type=7` means "strikethrough" in one implementation and "superscript" in another — the format has silently forked. Fixed-width records specifically prevent forward-compatible extension: you cannot add a "URL" field to a link span without breaking every existing reader's offset arithmetic into the spans table. This is the same mistake early binary formats made before the industry learned to use length-prefixed or tagged records. The Architect has optimized for read simplicity at the cost of any realistic evolution path. The atomic write-to-temp-then-rename strategy is sound on Linux with a local filesystem and a single writer. It is not sound on NFS, SMB shares, FAT32, or any filesystem that doesn't provide atomic rename semantics — which covers a meaningful fraction of real user environments including network drives, USB sticks, and some cloud-synced folders. More practically: the spec mandates complete re-serialization on every save, which means that for a 500MB document with tens of thousands of spans, every keystroke-triggered autosave is a full file rewrite. The Pragmatist waved this away as "fine for a single-user tool," but autosave intervals exist precisely because users don't manually save — and a format that makes autosave expensive will either have autosave disabled or will cause noticeable latency. This isn't a theoretical scaling cliff; it's the first complaint you'll see on the issue tracker from anyone working with moderately large documents.

⚙PragmatistR2

The Skeptic is right about the offset unit enforcement problem, but the proposed solution is impractical: you cannot spec your way to correctness across independent implementations. The real answer is to ship a reference parser library — which the Architect already proposed — and make it the canonical implementation that every editor wraps rather than reimplements. If the C library is the thing that reads and writes byte offsets, and editors are forbidden from doing their own span serialization, the off-by-one bug class is contained to one codebase. This is how you actually get interoperability in a young ecosystem: not through spec precision, but through a single trusted implementation. The type registry concern is also valid and has a boring solution — publish a central IANA-style registry document alongside the spec, reserve the top 256 values for standard types, and define the conforming behavior for unknown types as "render as unstyled text and preserve the record verbatim." Fixed-width records are fine if you freeze the record size and use a version field in the file header to introduce new record shapes. This is not elegant but it works and it's what virtually every successful binary format does. The autosave performance concern is the one I'd take most seriously as a practical matter. Full re-serialization on every save is fine when the document is small, which covers most use cases for a v1 — the median Markdown file is not 500MB. But the Skeptic is right that this becomes a real complaint fast. The fix is straightforward: don't autosave on every keystroke. Autosave on a timer with a dirty flag, say every 30 seconds or on focus loss. This is what every serious editor does anyway. For the rare user with genuinely large documents, you can add delta-serialization later without changing the on-disk format — it's purely an editor-side optimization. Don't over-engineer the format to solve a performance problem that affects a small minority of users in v1. The atomic rename concern about NFS and FAT32 is real but I'd explicitly descope it. Document in the spec that the format assumes a POSIX-compatible filesystem with atomic rename semantics, and note that network drives and FAT volumes are unsupported in v1. This is not a cop-out — it's the same boundary that SQLite draws, that Git draws, that virtually every tool making durability guarantees draws. Users on those environments already know they're in degraded territory. Trying to solve network filesystem consistency in a document format spec before you have a single user is a guaranteed way to never ship.

We document this explicitly: the format spec ships with a binding that says conforming implementations MUST use the reference library for span serialization, not reimplement it. This is how SQLite achieved byte-perfect interoperability across thirty years and thousands of applications. The off-by-one bug class the Skeptic describes exists only in implementations that go rogue, and we give them no reason to. Fixed-width 24-byte records are correct for the frozen core record shape, and the file header carries a format version that gates new record layouts. The "reference library as normative spec" strategy is the most dangerous assumption in this entire design, and it keeps being restated as though repetition makes it true. SQLite is not a valid analogy here. SQLite succeeded because it *is* the database — there is no meaningful distinction between the format and the library because the library is the only reasonable way to interact with the format. This document format is explicitly designed to coexist with plain text tools, editors, and pipelines. The moment you tell a developer that their Vim plugin, their Emacs mode, their Rust-based editor must FFI into a C library or be "out of conformance," you have not solved the ecosystem problem — you have replaced it with a worse one. Every non-C ecosystem will either ship a buggy reimplementation anyway or simply not support the format. The off-by-one bug class the Architect claims is "contained" is actually *guaranteed* the instant any second implementation exists, and implementations will exist because that's what developers do. The companion span solution for link URLs deserves serious scrutiny. A `link-target` record with identical start/end offsets is not an inelegant-but-workable solution — it's a latent correctness hazard. You now have a format where a single logical semantic unit (a hyperlink) is represented as two physically separate records that must be kept in sync, with no structural mechanism enforcing that relationship. What happens when a user deletes a span? Does the editor know to delete the companion? What if the spans table is sorted by offset and the two records end up non-adjacent after a sort? What if a tool strips "unknown" type records and silently drops all link targets while preserving the link spans? The spec can say "editors MUST handle companion spans atomically" but the format itself provides no enforcement, and this is precisely the category of invariant that breaks under real implementation pressure. You've taken the worst property of the original design — implicit coupling between separated records — and enshrined it as an architectural pattern. The dirty-flag autosave timer does not actually address the performance concern either; it merely defers it. A 30-second autosave on a 500MB document with tens of thousands of spans is still a full file rewrite happening in the background while the user is actively typing. On spinning rust, on a network-synced folder even with POSIX semantics, or on any system under I/O pressure, that write will block or stall the editor in ways users will notice. More importantly, the 30-second window is a 30-second data loss exposure on crash — which undermines the entire durability argument the format was making. The Pragmatist called this "fine for v1" but the spec is supposed to define conforming behavior, and a conforming editor that loses 30 seconds of work on crash is not actually delivering on the safety properties the CRC and atomic-rename machinery implies. The format is projecting durability guarantees it cannot keep.
